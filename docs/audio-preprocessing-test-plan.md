# 오디오 전처리 테스트 계획

이 문서는 회의 음성 전처리 전략을 결정하기 위한 테스트 계획이다.

기준일:

- 2026-04-29

목표:

1. 전처리 모드가 Cohere STT 정확도에 실제로 도움이 되는지 확인
2. 화자 분리 및 타임라인 정렬에 부작용이 없는지 확인
3. 기본값으로 둘 전처리 모드를 결정
4. 사용자 설정 UI 문구와 기본값을 근거 있게 정리

## 1. 비교 대상 모드

1차 비교 대상:

- `off`
- `auto`
- `loudnorm`
- `speechnorm` 후보

보류:

- `dynaudnorm`
  - 현재 짧은 저음량 샘플에서는 효과가 확인되지 않았다.
  - 자동 기본값이 아니라 실험 모드로만 유지한다.

- `silenceremove`
  - diarization / 타임스탬프 정렬에 미치는 영향이 커서 1차 비교 대상에서 제외한다.

- `denoise`
  - 한국어 자음/말끝 손실 위험이 있어 별도 실험으로 분리한다.

## 2. 테스트셋 구성

최소 1차 테스트셋:

- 총 6~10개 파일

권장 2차 테스트셋:

- 총 15~20개 파일

파일 유형:

1. 정상 음량, 조용한 환경
2. 작은 음량
3. 큰 음량
4. 화자 간 음량 차이가 큰 회의
5. 배경 소음 포함
   - 에어컨
   - 키보드
   - 마우스 클릭
   - 책상 충격음
6. 잔향/회의실 울림
7. 휴대폰 녹음
8. 긴 회의 파일
9. 한국어 고유명사/사내 용어 다수 포함
10. 영어/약어 혼용 발화 포함

샘플 길이 권장:

- 짧은 샘플: 30초 ~ 2분
- 중간 샘플: 3분 ~ 10분
- 긴 샘플: 10분 이상

## 3. 샘플별 메타데이터

각 테스트 파일에는 아래 메타데이터를 같이 기록한다.

- 파일명
- 길이
- 녹음 장치
- 녹음 장소
- 화자 수
- 예상 문제
  - 작은 음량
  - 잡음
  - 울림
  - 영어 혼용
  - 고유명사
- 원문 정답 유무
  - 있으면 참조 transcript 연결

## 4. 평가 항목

각 전처리 모드마다 아래를 비교한다.

1. STT 정확도
   - 한국어 고유명사 오인식
   - 영어/약어 오인식
   - 작은 목소리 누락 여부
   - 문장 경계 자연스러움

2. 화자 분리 영향
   - 화자 구간 경계 악화 여부
   - speaker alignment 어긋남 여부

3. 결과물 영향
   - 최종 summary 품질
   - action item / 결정사항 추출 품질

4. 처리 비용
   - 전처리 시간 증가량
   - 전체 분석 시간 증가량

## 4-1. 과거 문제 재확인 체크리스트

이 프로젝트에서 이미 실제로 문제였던 항목은 비교 테스트 때 반드시 다시 확인한다.

1. Cohere 경로의 timestamp 한계
   - Cohere는 현재 코드상 정확한 word/segment timestamp가 아니라 추정 timestamp를 사용한다.
   - 전처리 후에도 `timingApproximate` 기반 정렬 품질이 더 나빠지지 않는지 확인한다.

2. 화자 정렬 붕괴 여부
   - pyannote diarization 구간과 STT 문장 구간이 과도하게 어긋나지 않는지 본다.
   - 다화자 구간에서 문장 분할이 더 부자연스러워지지 않는지 확인한다.

3. 영어/로마자식 오인식
   - 기존 메모에서 일부 구간의 로마자/영어식 오인식이 남아 있었다.
   - 전처리가 이 문제를 줄이는지, 또는 오히려 키우는지 비교한다.

4. 비음성/저음량 구간 hallucination
   - Cohere 모델 카드는 non-speech에도 eager하게 전사할 수 있다고 명시한다.
   - 조용한 구간, 바닥잡음, 에어컨 소리에서 허위 전사가 늘지 않는지 확인한다.

5. 작은 목소리 손실
   - normalize/denoise/silence 처리로 작은 목소리나 말끝이 사라지지 않는지 본다.

6. 화자 분리 모델과의 상호작용
   - 전처리 후 diarization error가 커지지 않는지 확인한다.
   - overlap 구간에서 speaker assignment가 더 불안정해지지 않는지 본다.

7. 긴 파일 처리 안정성
   - chunking, 메모리 사용량, 처리 시간, 출력 일관성을 같이 본다.
   - 짧은 샘플에서는 좋아 보여도 긴 회의 파일에서 무너지지 않는지 확인한다.

8. 결과물 품질
   - transcript뿐 아니라 summary, action item, 결정사항 추출까지 같이 본다.
   - STT가 조금 좋아져도 요약이 흔들리면 기본값 채택을 다시 본다.

## 5. 평가 방식

정량 평가:

- 가능하면 샘플별 참조 transcript 준비
- WER/CER 또는 간단한 편집거리 비교

정성 평가:

- 고유명사
- 숫자
- 날짜
- 담당자명
- 영어 혼용 표현

권장 표:

```text
샘플 | 모드 | STT 품질 | 고유명사 | 작은 음량 | 화자정렬 | 요약 품질 | 처리시간 | 비고
```

## 6. 통과 기준

기본값 후보가 되려면 아래를 만족해야 한다.

1. 전체 샘플에서 `off` 대비 의미 있는 품질 향상 또는 최소 동등
2. 화자 분리 품질 저하가 없어야 함
3. 긴 파일에서 처리시간 증가가 과도하지 않아야 함
4. 특정 샘플에서만 좋아지고 전체적으로 불안정하면 기본값으로 채택하지 않음
5. 과거 문제 재확인 체크리스트에서 치명적인 회귀가 없어야 함

## 7. 실행 순서

1. 비교는 집 PC 등 성능이 충분한 환경에서 진행한다.
2. 샘플 폴더를 모은 뒤 Codex가 `scripts/analyze_audio_inventory.py`로 인벤토리 CSV를 생성한다.
3. Codex가 `docs/audio-testset-manifest-template.csv`를 기준으로 실제 manifest를 채운다.
4. 부족한 유형 샘플 수집
5. `off / auto / loudnorm` 1차 비교
6. 필요 시 `speechnorm` 추가
7. 승자 모드를 기본값 후보로 지정
8. 그 결과를 UI 설정 문구와 도움말에 반영
