# LMO 스마트 회의시스템 Design System

## 원칙

- 첫 화면은 회의록 작성과 최근 기록 확인에 집중한다.
- 사용자 화면에는 모델명, 파이프라인, 서버 같은 내부 용어를 최소화한다.
- 반복 UI는 `index.css`의 공통 클래스를 우선 사용한다.
- 카드 반경은 작게 유지하고, 정보 밀도는 업무 도구에 맞게 차분하게 유지한다.

## 핵심 용어

- `회의 요약`: 분석 후 핵심 내용, 주요 주제, 할 일을 보여주는 영역
- `대화록`: 화자, 시간, 문장을 함께 보여주는 영역
- `분석 준비`: 모델 또는 로컬 분석 기능 확인 단계
- `음성 파일`: 음성 파일과 영상 파일을 모두 포함하는 업로드 표현. 영상도 분석 대상이지만 사용자는 최종적으로 음성을 분석한다고 이해한다.
- `발화자 구분`: 누가 말했는지 자동으로 나누는 분석 옵션

## 공통 클래스

- `app-panel`: 주요 작업 패널
- `app-panel-header`: 패널 상단 정보 영역
- `tab-list`, `tab-button`, `tab-button-active`: 탭 UI
- `summary-block`: 요약 본문
- `topic-chip`: 주요 주제 태그
- `speaker-turn`, `speaker-meta`: 대화록 행
- `status-note`: 짧은 안내 및 주의 문구

## 설정 화면 기준

- 분석 준비 탭은 준비 상태와 필요한 파일을 `models` 폴더에 넣는 짧은 안내만 보여준다.
- 분석 탭은 기본값을 유지하되, 필요할 때 조정하는 옵션만 둔다.
- 기능 없는 정보 탭은 두지 않는다. 사용자가 조정하거나 확인할 수 있는 설정만 남긴다.

## 피해야 할 표현

- 사용자 화면에서 특정 모델명으로 기능을 설명하지 않는다.
- `SSE`, `FastAPI`, `diarization pipeline` 같은 구현 용어를 일반 화면에 노출하지 않는다.
- 반복 화면에 긴 사용 설명을 고정 배치하지 않는다.

## 색상 사용 규칙

- 화면 컴포넌트에서 `bg-blue-*`, `bg-red-*`, `bg-amber-*`, `text-blue-*`, `border-red-*` 같은 원시 색상 유틸을 직접 조합하지 않는다.
- 상태 안내는 `StatusBanner`와 `ProgressBar`를 우선 사용한다.
- 상태 색상은 `index.css`의 semantic status token을 기준으로 한다: `info`, `warning`, `error`, `success`, `neutral`.
- 버튼, 카드, 메뉴, 빈 상태처럼 반복되는 UI는 색상/테두리/hover를 화면에서 직접 만들지 않고 공통 클래스나 typed component prop으로 지정한다.
- 컴포넌트 호출부에서는 배치, 간격, 반응형 정도만 조합하고 색상 결정은 디자인 시스템에 둔다.
- 화자 구분처럼 여러 색이 필요한 경우에는 예외적으로 데이터 구분 팔레트를 사용하되, `speaker-*` 계열 공통 클래스 안에서만 정의한다.
