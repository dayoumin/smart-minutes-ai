# 히스토리 오른쪽 본문 UX 개선 계획

## 목적

히스토리 화면의 오른쪽 본문을 “선택된 회의 하나를 검토하고, 정리하고, 회의록으로 만드는 작업 공간”으로 재정의한다.

현재 방향은 다음 원칙을 기준으로 한다.

- 왼쪽 영역은 회의 찾기와 선택에 집중한다.
- 오른쪽 본문은 선택된 회의의 내용 확인, AI 정리, 양식 회의록 생성을 담당한다.
- 반복 화면에는 긴 설명을 두지 않고, 사용자가 바로 판단하거나 실행할 수 있는 정보만 둔다.
- AI 결과는 확정본이 아니라 검토 가능한 초안으로 보여준다.

## 현재 문제

### 선택 흐름

- 사용자가 `이전 회의 기록`을 클릭했을 때 오른쪽 본문에서 다시 회의를 고르는 구조가 생기면 “내부에 또 내부”처럼 보인다.
- 오른쪽 본문은 회의 목록과 상세를 동시에 다루기보다, `목록 화면`과 `상세 화면`을 분리하는 편이 자연스럽다.
- 특정 회의 선택은 히스토리 목록 화면에서 끝나고, 선택 이후에는 해당 회의 상세만 보여준다.

### 정보 밀도

- 상단에 제목, 일시, 참석자, 원본 파일, 다운로드 형식, 다운로드, 수정, 삭제, 탭, 검색이 몰리면 관리 도구처럼 보인다.
- 원본 파일명 같은 보조 정보는 접거나 작은 메타 정보로 낮춰도 된다.
- “몇 개 저장됨” 같은 관리성 문구는 주요 콘텐츠보다 작게 다뤄야 한다.

### 용어

- 사용자 화면에서는 프로젝트 기준 용어를 유지한다.
- 기본 탭 이름은 `회의 요약`, `발화 기록`을 우선 사용한다.
- `대화 내용`은 친숙하지만 기존 디자인 문서와 맞지 않으므로, 문서/화면 용어를 통일하기 전까지는 `발화 기록`을 기준으로 한다.

## 목표 화면 구조

### 상단 영역

상단은 선택된 회의의 정체성과 핵심 액션만 보여준다.

- 회의 제목
- 일시
- 참석자
- 다운로드
- 수정
- 삭제

보조 정보:

- 원본 파일명
- 저장된 회의 수
- 분석 품질 경고

보조 정보는 상단에 크게 노출하지 않고, 작은 메타 줄이나 접힘 영역으로 둔다.

### 본문 탭

초기 탭 구성:

- `회의 요약`
- `발화 기록`
- `AI 정리`
- `양식 회의록`

탭은 네 개를 넘기지 않는다. 추가 기능은 탭 안의 액션이나 메뉴로 넣는다.

### 검색

- 검색은 오른쪽 본문 전체 검색보다 현재 회의 안에서만 동작하게 한다.
- 검색창 문구는 `현재 회의에서 검색`으로 둔다.
- 검색 결과가 없을 때는 짧게 `검색 결과가 없습니다.`만 보여준다.

## 회의 요약 탭

회의 요약 탭은 단순 텍스트 요약이 아니라 업무 판단에 필요한 구조로 재배치한다.

권장 섹션:

- 핵심 요약
- 결정 사항
- 할 일
- 쟁점 및 미결 사항
- 확인 필요 사항
- 주요 주제

초기 구현에서는 이미 저장된 `summary`, `topics`, `actions`를 사용한다. `decisions`, `needs_check`는 백엔드 요약 구조와 저장 모델을 맞춘 뒤 추가한다.

## 발화 기록 탭

발화 기록 탭은 원문 확인과 근거 추적을 담당한다.

기본 구성:

- 화자
- 시간
- 발화 내용
- 시간 추정 표시
- 인식 품질 확인 표시

후속 개선:

- 화자별 필터
- 참석자 실명 매핑
- AI 요약 문장에서 관련 발화로 이동
- 긴 발화 기록의 섹션 접기

## AI 정리 탭

AI 정리 탭은 회의 내용을 다시 업무용 산출물로 가공하는 곳이다.

MVP 출력:

- 참석자별 요약
- 결정 사항
- 할 일
- 논의된 쟁점
- 확인 필요 항목

중요 정책:

- AI 정리는 자동 확정본이 아니라 초안이다.
- 기존 회의록을 즉시 덮어쓰지 않는다.
- 결과는 미리보기로 보여주고 사용자가 `적용`해야 저장한다.
- 재생성 결과와 사용자 수정본을 분리한다.

초기 액션:

- `AI 정리 생성`
- `다시 생성`
- `현재 회의록에 적용`

피해야 할 것:

- 채팅형 도구를 먼저 넣는 것
- 버튼을 지나치게 많이 나열하는 것
- 모델명, 서버, 파이프라인 같은 구현 용어를 보여주는 것

## 양식 회의록 탭

양식 회의록은 사용자가 미리 정한 구조에 맞춰 AI가 회의 내용을 재정리하는 기능이다.

MVP는 기본 양식 1개로 시작한다.

### 기본 회의록 양식

- 회의명
- 일시
- 참석자
- 안건
- 논의 내용
- 결정 사항
- 할 일
- 발화 기록 첨부 여부

후속 양식 후보:

- 보고용 요약
- 참석자별 정리
- 의사결정 기록
- 고객/외부 미팅
- 기관/부서별 회의록

양식 기능의 핵심 데이터:

- 양식 정의
- AI 생성 결과
- 사용자 편집본
- 내보내기 포맷

## 구현 순서

### 1단계: 히스토리 선택 모델 정리

목표:

- 왼쪽은 회의 선택, 오른쪽은 선택된 회의 상세라는 역할을 명확히 한다.
- 오른쪽 본문 안의 중복 회의 목록이나 드롭다운은 제거하거나 모바일 대체 UI로 제한한다.

작업:

- 히스토리 목록에서 특정 회의를 선택하는 흐름 확정
- `onSelectMeeting` 사용 방식 정리
- 기본 자동 선택 정책 재검토
- 빈 상태 문구 정리

### 2단계: 오른쪽 본문 정보 밀도 축소

목표:

- 상단을 읽기 쉬운 회의 상세 헤더로 만든다.
- 보조 정보와 관리 문구를 낮춘다.

작업:

- 회의 제목/일시/참석자/핵심 액션만 상단에 유지
- 원본 파일명은 보조 메타로 이동
- 검색은 현재 회의 기준으로 유지
- 탭 용어를 `회의 요약`, `발화 기록`으로 통일

### 3단계: 회의 요약 구조 개선

목표:

- 요약 결과를 업무 판단에 바로 쓰게 만든다.

작업:

- 핵심 요약, 주요 주제, 할 일을 명확히 분리
- 백엔드의 `decisions`, `needs_check` 필드를 UI에 연결할 준비
- 빈 섹션은 숨김 처리

### 4단계: AI 정리 MVP

목표:

- 기존 발화 기록을 기반으로 참석자별 요약과 후속 작업 정리를 만든다.

작업:

- AI 정리 결과 저장 모델 추가
- 참석자별 요약 생성
- 결정 사항, 쟁점, 확인 필요 항목 생성
- 미리보기 후 적용 흐름 추가

### 5단계: 양식 회의록 MVP

목표:

- 기본 회의록 양식으로 AI가 구조화된 회의록 초안을 만든다.

작업:

- 기본 양식 정의
- 양식 선택 UI
- 생성 미리보기
- 적용/다운로드 흐름
- HWPX/DOCX 내보내기와 연결 검토

## 리스크

- 발화자 구분 품질이 낮으면 참석자별 요약 신뢰도가 떨어진다.
- AI 결과를 확정본처럼 보여주면 회의록 책임 소재가 불명확해진다.
- AI 재생성이 사용자 수정본을 덮어쓰면 신뢰를 잃는다.
- 양식 기능을 처음부터 많이 열면 저장, 편집, 내보내기 정책이 복잡해진다.
- 긴 파일에서는 AI 정리 시간, 실패, 재시도 UX가 중요하다.

## UX 원칙

- 반복 화면에는 긴 설명을 두지 않는다.
- 사용자는 먼저 회의 내용을 읽고, 필요할 때 AI 정리를 실행한다.
- AI 기능은 “도움 도구”로 보이게 하고 자동 완성처럼 과장하지 않는다.
- 중요한 결과는 항상 사용자가 검토하고 적용한다.
- 화면 용어는 `회의 파일`, `회의 요약`, `발화 기록`, `분석 준비` 등 사용자 친화적 표현을 우선한다.
