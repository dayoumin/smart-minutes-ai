# 히스토리 오른쪽 본문 UX 개선 계획

## 목적

히스토리 화면의 오른쪽 본문을 “선택된 회의 하나를 검토하고, 정리하고, 회의록으로 만드는 작업 공간”으로 재정의한다.

현재 방향은 다음 원칙을 기준으로 한다.

- 왼쪽 영역은 회의 찾기와 선택에 집중한다.
- 오른쪽 본문은 선택된 회의의 내용 확인, AI 정리, 양식 회의록 생성을 담당한다.
- 반복 화면에는 긴 설명을 두지 않고, 사용자가 바로 판단하거나 실행할 수 있는 정보만 둔다.
- AI 결과는 확정본이 아니라 검토 가능한 초안으로 보여준다.

## 현재 문제

### 선택 흐름

- 사용자가 `이전 회의 기록`을 클릭했을 때 오른쪽 본문에서 다시 회의를 고르는 구조가 생기면 “내부에 또 내부”처럼 보인다.
- 오른쪽 본문은 회의 목록과 상세를 동시에 다루기보다, `목록 화면`과 `상세 화면`을 분리하는 편이 자연스럽다.
- 특정 회의 선택은 히스토리 목록 화면에서 끝나고, 선택 이후에는 해당 회의 상세만 보여준다.

### 정보 밀도

- 상단에 제목, 일시, 참석자, 원본 파일, 다운로드 형식, 다운로드, 수정, 삭제, 탭, 검색이 몰리면 관리 도구처럼 보인다.
- 원본 파일명 같은 보조 정보는 접거나 작은 메타 정보로 낮춰도 된다.
- “몇 개 저장됨” 같은 관리성 문구는 주요 콘텐츠보다 작게 다뤄야 한다.

### 용어

- 사용자 화면에서는 프로젝트 기준 용어를 유지한다.
- 기본 탭 이름은 `회의 요약`, `대화록`을 우선 사용한다.
- `대화록`은 원문 확인 목적이 직관적으로 드러나고, `발화 기록`보다 일반 사용자에게 이해하기 쉽다.

## 목표 화면 구조

### 상단 영역

상단은 선택된 회의의 정체성과 핵심 액션만 보여준다.

- 회의 제목
- 일시
- 참석자
- 다운로드
- 수정
- 삭제

보조 정보:

- 원본 파일명
- 저장된 회의 수
- 분석 품질 경고

보조 정보는 상단에 크게 노출하지 않고, 작은 메타 줄이나 접힘 영역으로 둔다.

### 본문 탭

초기 탭 구성:

- `회의 요약`
- `대화록`
- `AI 정리`
- `양식 회의록`

탭은 네 개를 넘기지 않는다. 추가 기능은 탭 안의 액션이나 메뉴로 넣는다.

### 검색

- 검색은 오른쪽 본문 전체 검색보다 현재 회의 안에서만 동작하게 한다.
- 검색창 문구는 `현재 회의에서 검색`으로 둔다.
- 검색 결과가 없을 때는 짧게 `검색 결과가 없습니다.`만 보여준다.

## 회의 요약 탭

회의 요약 탭은 단순 텍스트 요약이 아니라 업무 판단에 필요한 구조로 재배치한다.

권장 섹션:

- 핵심 요약
- 결정 사항
- 할 일
- 쟁점 및 미결 사항
- 확인 필요 사항
- 주요 주제

초기 구현에서는 이미 저장된 `summary`, `topics`, `actions`를 사용한다. 이후 AI 요약 결과에 `topic_sections`, `participant_summaries`, `decisions`, `needs_check`를 포함해 주제별 내용과 발언자별 요약을 함께 보여준다.

## 대화록 탭

대화록 탭은 원문 확인과 근거 추적을 담당한다.

기본 구성:

- 화자
- 시간
- 발화 내용
- 시간 추정 표시
- 인식 품질 확인 표시

후속 개선:

- 화자별 필터
- 참석자 실명 매핑
- AI 요약 문장에서 관련 발화로 이동
- 긴 대화록의 섹션 접기

## AI 정리 탭

AI 정리 탭은 회의 내용을 다시 업무용 산출물로 가공하는 곳이다.

MVP 출력:

- 발언자별 요약
- 결정 사항
- 할 일
- 논의된 쟁점
- 확인 필요 항목

권장 생성 흐름:

- 음성 인식과 화자 구분을 먼저 끝내고 대화록을 저장한다.
- 저장된 대화록을 기반으로 주제별 내용, 발언자별 요약, 결정 사항, 할 일, 확인 필요 항목을 AI가 별도로 생성한다.
- 분석 직후 자동 생성할 수 있지만, 사용자가 대화록 품질을 확인한 뒤 `AI 정리 생성` 또는 `다시 생성`을 누를 수 있어야 한다.
- 사용자가 `발언자 1` 같은 화자 라벨을 실제 참석자 이름으로 매핑하면, 발언자별 요약은 이름 기준으로 다시 표시하거나 재생성할 수 있어야 한다.
- 음성 인식 품질이 낮은 회의록은 AI 정리 결과에도 품질 경고를 함께 보여준다.

중요 정책:

- AI 정리는 자동 확정본이 아니라 초안이다.
- 기존 회의록을 즉시 덮어쓰지 않는다.
- 결과는 미리보기로 보여주고 사용자가 `적용`해야 저장한다.
- 재생성 결과와 사용자 수정본을 분리한다.

초기 액션:

- `AI 정리 생성`
- `다시 생성`
- `현재 회의록에 적용`

피해야 할 것:

- 채팅형 도구를 먼저 넣는 것
- 버튼을 지나치게 많이 나열하는 것
- 모델명, 서버, 파이프라인 같은 구현 용어를 보여주는 것

## 양식 회의록 탭

양식 회의록은 사용자가 미리 정한 구조에 맞춰 AI가 회의 내용을 재정리하는 기능이다.

MVP는 기본 양식 1개로 시작한다.

### 기본 회의록 양식

- 회의명
- 일시
- 참석자
- 안건
- 논의 내용
- 결정 사항
- 할 일
- 대화록 첨부 여부

후속 양식 후보:

- 보고용 요약
- 참석자별 정리
- 의사결정 기록
- 고객/외부 미팅
- 기관/부서별 회의록

양식 기능의 핵심 데이터:

- 양식 정의
- AI 생성 결과
- 사용자 편집본
- 내보내기 포맷

## 구현 순서

### 1단계: 히스토리 선택 모델 정리

목표:

- 히스토리 진입 시 목록 화면을 먼저 보여준다.
- 기록을 선택하면 상세 화면으로 전환하고, 상세 화면 안에서는 다른 회의 선택 UI를 반복하지 않는다.
- 데스크톱과 모바일에서 동일하게 목록에서 상세로 들어가는 흐름을 유지한다.
- 사이드바에는 최근/고정 회의록을 최대 10개까지 빠르게 보여주되, 전체 기록으로 확장할 수 있어야 한다.

작업:

- 히스토리 목록에서 특정 회의를 선택하는 흐름 확정
- 일반 히스토리 진입에서는 최신 회의를 자동 선택하지 않는다.
- 새 분석이 완료된 직후에는 방금 생성된 회의 상세를 바로 열 수 있다.
- 10개 밖의 오래된 회의록이 UI에서 사라지지 않도록 전체 보기 또는 검색 진입점을 둔다.
- 빈 상태 문구 정리

### 2단계: 오른쪽 본문 정보 밀도 축소

목표:

- 상단을 읽기 쉬운 회의 상세 헤더로 만든다.
- 보조 정보와 관리 문구를 낮춘다.

작업:

- 회의 제목/일시/참석자/핵심 액션만 상단에 유지
- 원본 파일명은 보조 메타로 이동
- 검색은 현재 회의 기준으로 유지
- 탭 용어를 `회의 요약`, `대화록`으로 통일

### 3단계: 회의 요약 구조 개선

목표:

- 요약 결과를 업무 판단에 바로 쓰게 만든다.

작업:

- 핵심 요약, 주요 주제, 할 일을 명확히 분리
- 주요 주제 칩은 AI가 만든 주제별 내용을 열어보는 진입점으로 사용
- 발언자별 요약 섹션 추가
- 백엔드의 `decisions`, `needs_check` 필드를 UI에 연결할 준비
- 빈 섹션은 숨김 처리

### 4단계: AI 정리 MVP

목표:

- 기존 대화록을 기반으로 발언자별 요약과 후속 작업 정리를 만든다.

작업:

- AI 정리 결과 저장 모델 추가
- 발언자별 요약 생성
- 화자 라벨을 실제 참석자 이름으로 매핑하는 흐름 추가
- 결정 사항, 쟁점, 확인 필요 항목 생성
- 미리보기 후 적용 흐름 추가

### 5단계: 양식 회의록 MVP

목표:

- 기본 회의록 양식으로 AI가 구조화된 회의록 초안을 만든다.

작업:

- 기본 양식 정의
- 양식 선택 UI
- 생성 미리보기
- 적용/다운로드 흐름
- HWPX/DOCX 내보내기와 연결 검토

## 리스크

- 발화자 구분 품질이 낮으면 발언자별 요약 신뢰도가 떨어진다.
- AI 결과를 확정본처럼 보여주면 회의록 책임 소재가 불명확해진다.
- AI 재생성이 사용자 수정본을 덮어쓰면 신뢰를 잃는다.
- 양식 기능을 처음부터 많이 열면 저장, 편집, 내보내기 정책이 복잡해진다.
- 긴 파일에서는 AI 정리 시간, 실패, 재시도 UX가 중요하다.

## UX 원칙

- 반복 화면에는 긴 설명을 두지 않는다.
- 사용자는 먼저 회의 내용을 읽고, 필요할 때 AI 정리를 실행한다.
- AI 기능은 “도움 도구”로 보이게 하고 자동 완성처럼 과장하지 않는다.
- 중요한 결과는 항상 사용자가 검토하고 적용한다.
- 화면 용어는 `음성 파일`, `회의 요약`, `대화록`, `분석 준비` 등 사용자 친화적 표현을 우선한다.

## 1차/2차/3차 AI 정리 생성 구조

### 목적

음성 인식 직후 모든 AI 정리를 한 번에 확정 결과처럼 만들지 않는다. 먼저 기본 회의록을 안정적으로 저장하고, 주제별 정리와 발언자별 맥락 정리는 별도 생성 단계로 분리한다. 특히 발언자별 정리는 단순히 특정 화자의 발언만 모아 요약하는 기능이 아니라, 전체 회의 맥락과 주제별 흐름 안에서 해당 화자의 입장, 반응, 질문, 책임을 정리하는 기능으로 본다.

### 1차: 기본 회의록

분석 직후 바로 생성하고 저장한다. 현재 운영 기준에서는 기본 분석 완료를 대화록 저장으로 본다. 요약 AI가 준비되지 않았거나 `summary.generate_during_analysis=false`이면 전체 회의 요약은 자동 생성하지 않고, `generationStatus.summary=skipped` 또는 `not_started` 상태로 남긴 뒤 회의 기록 상세 화면에서 사용자가 별도로 실행한다.

- 음성 인식 결과
- 화자 구분 결과
- 대화록
- 전체 회의 요약: 선택 생성
- 주요 주제 목록: 선택 생성
- 결정 사항: 선택 생성
- 할 일: 선택 생성
- 확인 필요 사항: 선택 생성

1차 결과는 회의록의 기본 뼈대다. 사용자는 분석 완료 후 이 기본 회의록 상세 화면으로 이동하거나 `결과 보기` 액션을 통해 진입할 수 있어야 한다. 이 단계에서 사용자에게 "회의록 생성 완료"라고 말할 때는 대화록 기준 완료인지, 요약까지 완료인지 UI에서 분리해서 보여준다.

### 2차: 주제별 정리

기본 회의록 저장 이후 별도 생성한다. 자동 후속 생성으로 시작할 수 있지만, 사용자가 명시적으로 `주제별 정리 만들기`를 실행할 수 있는 구조를 우선한다.

- 각 주제의 논의 내용
- 관련 근거
- 쟁점
- 관련 결정 사항
- 관련 할 일

2차 생성 입력은 전체 대화록과 1차 기본 회의록이다. `topicSections`가 있으면 주제 정리의 주 데이터로 사용하고, 단순 `topics` 목록은 색인 또는 fallback으로 낮춘다.

완료 기준:

- 3~7개 주제 섹션을 우선 목표로 한다.
- `핵심 주제`처럼 한 덩어리 요약만 반환되면 재시도하거나 사용자에게 다시 정리할 수 있게 한다.
- 각 주제는 `topic`, `summary`, `evidence`, `actions`를 포함한다.
- 긴 회의에서는 시간대별/부분 요약 결과를 함께 사용해 중간 쟁점이 누락되지 않게 한다.

### 3차: 발언자별 맥락 정리

주제별 정리 이후 별도 생성한다. 발언자별 정리는 `화자 1이 말한 문장 요약`이 아니라, 전체 회의 맥락에서 해당 화자가 어떤 역할을 했는지 정리한다.

- 어떤 주제에 관여했는지
- 어떤 입장이나 제안을 냈는지
- 누구의 의견에 반응했는지
- 동의, 반대, 질문, 보완, 결정 제안이 있었는지
- 맡은 일이나 확인할 일이 있는지

실제 참석자 이름과 자동 화자 라벨은 분리한다. 이름 매핑 전에는 `발언자별 AI 초안` 또는 `화자 라벨 기준 정리`로 표시하고, 실제 참석자별 확정 정리처럼 보여주지 않는다.

생성 입력은 특정 화자의 발언 목록만으로 제한하지 않는다. 각 화자의 발언 묶음에 직전/직후 발언, 관련 주제 섹션, 결정 사항, 할 일을 함께 넣어야 한다. 그래야 참석자가 어떤 맥락에서 질문했는지, 반대한 것인지 보완한 것인지, 결정에 책임이 있는지 판단할 수 있다.

### 데이터 모델 방향

기존 필드와 호환하되, 향후 확장을 위해 생성 상태와 화자 매핑을 분리한다.

- `summary`: 1차 기본 회의록 요약
- `topics`: 1차 주요 주제 목록
- `decisions`: 1차 결정 사항
- `actions`: 1차 할 일
- `needsCheck`: 1차 확인 필요 사항
- `topicSections`: 2차 주제별 정리
- `speakerContextSummaries`: 3차 발언자별 맥락 정리
- `participantSummaries`: legacy 호환 또는 3차 결과 표시용 alias로만 사용
- `generationStatus`: `summary`, `topicSections`, `speakerContextSummaries` 각각의 생성 상태
- `speakerMap`: `speakerId`, `displayName`, `participantId` 매핑

정합성 기준:

- 현재 구현에서는 `summary`를 새로 만들면 기존 `topicSections`, `speakerContextSummaries`, `participantSummaries`를 초기화하고 재생성 필요 상태로 되돌린다.
- 회의 제목, 목적/설명, 대화록이 바뀌면 관련 정리 결과는 stale 또는 재생성 필요 상태로 표시한다.
- 화자 이름 매핑 변경은 현재 관련 정리 결과를 stale 처리하지 않고 표시명 동기화만 한다. 참석자별 정리를 다시 생성해야 하는 정책이 필요하면 별도 구현한다.
- 전체 요약과 주제별 정리는 서로 다른 결론을 내면 안 된다. 전체 요약은 가능하면 주제 섹션을 기준으로 재생성하거나, 주제 섹션과의 차이를 검증한다.
- 입력 fingerprint 기준은 프론트 IndexedDB record와 백엔드 output JSON에서 같은 의미로 유지한다.

### API 방향

1차 분석 API는 기본 회의록을 만든다. 2차/3차는 저장된 분석 결과를 대상으로 별도 생성 API를 둔다.

현재 구조에서는 서버에 별도 회의 DB가 없고, 백엔드는 `job_id` 기준 output JSON을 알고 있으며 프론트는 IndexedDB의 `MeetingRecord`를 알고 있다. 따라서 1차 구현에서는 `meeting id`보다 `job_id` 기반 API가 더 정직하다.

- `POST /api/analyze`: 1차 기본 회의록 생성
- `POST /api/outputs/{job_id}/generate-topic-sections`: 2차 주제별 정리 생성
- `POST /api/outputs/{job_id}/generate-topic-section`: 사용자 지정 단일 주제 추가 정리 생성
- `POST /api/outputs/{job_id}/generate-speaker-context`: 3차 발언자별 맥락 정리 생성

단일 주제 추가 정리는 사용자가 특정 주제를 보강하는 기능이다. 전체 주제별 정리 완료 상태를 대체하지 않으며, 주제 섹션이 충분하지 않으면 참석자별 맥락 정리의 전제조건으로 보지 않는다.

IndexedDB에만 남아 있거나 legacy record처럼 백엔드 output JSON과 연결되지 않은 기록을 위해서는 full meeting payload를 받는 보조 API를 둘 수 있다.

- `POST /api/generate-topic-sections`
- `POST /api/generate-speaker-context`

2차/3차 API는 생성 중, 완료, 실패, 취소 상태를 UI가 표시할 수 있도록 상태 값을 반환해야 한다. 생성 성공 후에는 백엔드 output JSON과 프론트 IndexedDB의 상태 동기화 정책을 명확히 해야 한다.

### UI 방향

- 분석 완료 후 기본 회의록 상세로 이동하거나 `결과 보기`를 명확히 제공한다.
- 요약 탭에는 1차 기본 회의록을 먼저 보여준다.
- 주제별 정리가 없으면 `주제별 정리 만들기` 액션을 보여준다.
- 발언자별 맥락 정리가 없으면 `발언자별 맥락 정리 만들기` 액션을 보여준다.
- 발언자별 결과에는 항상 `AI 초안`, `화자 라벨 기준`, `검토 필요` 성격을 표시한다.
- 화자 구분 실패, `UNKNOWN`, 단일 화자, 시간 추정 다수, 오인식 의심 다수인 경우 발언자별 정리를 강조하지 않고 경고와 함께 보여준다.

### 상세 화면 정보 위계

요약 탭은 사용자가 회의 내용을 빠르게 파악한 뒤 필요한 AI 정리를 추가로 만들 수 있게 한다. 상단부터 `핵심 요약`, `추가 정리`, `주제별 정리`, `발언자별 맥락 정리`, `할 일` 순서로 배치한다.

- `핵심 요약`: 1차 분석 결과의 가장 짧은 결론
- `추가 정리`: 주제별/발언자별 생성 상태와 실행 버튼. 상태를 과하게 강조하지 않고, 사용자가 실행할 수 있는 작업으로 보여준다.
- `주요 내용 및 결정사항`: 결정사항, 논의 내용, 확인 필요 항목을 함께 확인하는 영역. 긴 주제명은 화면에서 축약하고 상세 내용은 선택 후 확인한다.
- `발언자별 맥락 정리`: 주제별 정리 이후 생성되는 AI 초안 영역
- `할 일`: 사용자가 바로 확인해야 하는 후속 조치

발언자별 맥락 정리는 주제별 정리가 없으면 `대기`로 표시하고, 버튼은 비활성화한다. 이 규칙은 프론트와 백엔드 모두에서 동일하게 강제한다.

### 내보내기 방향

내보내기는 사용자가 더 공식 문서처럼 받아들이므로 UI보다 보수적으로 표시한다.

- 2차/3차 미생성 섹션은 비워두거나 `생성되지 않음`으로 표시한다.
- 발언자별 맥락 정리는 `AI 초안`과 `검토 필요` 문구를 포함한다.
- MD/DOCX/HWPX는 동일한 섹션 정책을 따른다.

### 구현 순서

1. 주제별 정리가 안정적인 회의 구조를 만들도록 품질 기준을 고정한다.
2. 전체 요약과 주제별 정리의 입력/무효화/재생성 기준을 맞춘다.
3. 발언자별 맥락 정리는 화자 발언 목록이 아니라 맥락 묶음 기준으로 생성한다.
4. 화자 이름 매핑 모델을 정리하고, 이름 매핑 전/후 표시 용어를 분리한다.
5. export 섹션 정책을 공통화하고 테스트를 추가한다.
