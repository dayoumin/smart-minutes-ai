# 긴 영상 자동 처리 및 사용자 안내 계획

기준일: 2026-06-03

## 목적

긴 영상이나 긴 음성 파일도 사용자가 기술적인 처리 방식을 고르지 않고 회의록을 만들 수 있게 한다. 앱은 내부적으로 파일 길이, 임시 파일 크기, 참석자 구분 위험도, 이어하기 가능성을 판단하고 필요한 경우에만 사용자에게 선택을 요청한다.

사용자에게 필요한 정보는 다음 네 가지다.

- 지금 무엇을 하고 있는가
- 시간이 많이 걸릴 수 있는가
- 중간에 실패해도 무엇이 남는가
- 사용자가 지금 해야 할 일이 있는가

## 사용자 관점 원칙

1. 기본은 자동 진행이다.
   - 사용자가 `긴 파일 모드`, `청크`, `checkpoint`, `diarization` 같은 용어를 직접 이해할 필요가 없어야 한다.
   - 파일이 길면 앱이 자동으로 긴 파일 처리 방식을 적용한다.

2. 대화록은 먼저 살린다.
   - 긴 파일에서 가장 중요한 결과물은 대화록이다.
   - 참석자 구분이나 정리가 실패해도 대화록은 회수할 수 있어야 한다.

3. 오래 걸리면 오래 걸린다고 말한다.
   - 처음부터 정확한 완료 시간을 약속하지 않는다.
   - 진행하면서 추정치를 갱신한다.
   - 참석자 구분과 정리는 대화록 작성보다 오래 걸릴 수 있음을 분리해서 알려준다.

4. 사용자가 선택해야 하는 순간은 예외 상황으로 제한한다.
   - 저장 공간 부족
   - 원본 음성 파일 없음
   - 참석자 구분 반복 실패
   - 현재 PC에서 참석자 구분이 위험한 길이
   - 앱 재시작 후 이어하기 여부 확인

## 현재 이미 갖춘 기반

현재 앱에는 긴 영상 처리를 뒷받침하는 기반이 일부 있다.

- 파일 선택 시 파일 종류, 크기, 길이를 표시한다.
- 대화록 작성 단계는 30초 단위 음성 구간으로 나눠 처리한다.
- 분석 진행 화면에서 경과 시간과 대화록 저장까지의 예상 시간을 표시한다.
- 분석 중 이동/닫기 방지 흐름이 있다.
- 중지와 취소를 구분한다.
- 같은 파일을 다시 선택하면 이전 음성 인식 진행분을 이어서 쓸 수 있는 흐름이 있다.
- 결과 화면에서 대화록, 참석자 구분, 기록 정리 상태가 분리되어 있다.
- 참석자 구분은 결과 화면에서 별도로 실행할 수 있다.
- 참석자 구분 중지/취소 UI가 있다.
- 참석자 구분이 제외되면 대화록은 그대로 사용할 수 있다는 안내가 있다.
- 설정에 내부 음성 파일 보존 옵션이 있어 나중에 참석자 구분을 다시 실행할 수 있다.
- 백엔드에는 STT chunk checkpoint, diarization checkpoint 재사용, source.wav 보존 정책이 있다.

## 현재 부족한 부분

1. 파일 선택 직후의 긴 파일 안내가 약하다.
   - 길이는 보이지만, 그 길이가 처리 시간과 참석자 구분 위험에 어떤 의미인지 알려주지 않는다.

2. 대화록 예상 시간과 전체 작업 시간이 분리되어 있지만, 사용자가 쉽게 이해하기에는 설명이 약하다.
   - 현재 예상 시간은 대화록 저장 기준이다.
   - 참석자 구분과 기록 정리는 별도 시간이 걸린다는 안내를 더 자연스럽게 보여줘야 한다.

3. PC 상태 점검이 부족하다.
   - RAM 여유, 디스크 여유, 절전 상태, 배터리 상태 같은 사용 환경 안내가 없다.
   - 최소한 디스크 여유는 긴 영상 시작 전 확인해야 한다.

4. 참석자 구분 위험도 안내가 늦다.
   - 현재는 실행 시점에 너무 길거나 크면 제외한다.
   - 사용자는 파일 선택 단계에서 “대화록은 가능하지만 참석자 구분은 오래 걸릴 수 있음”을 알아야 한다.

5. 이어하기가 가능한 상태와 불가능한 상태를 더 명확히 보여줘야 한다.
   - “중간 결과가 저장됨”, “같은 파일을 선택하면 이어서 진행 가능”, “원본 음성이 없어 참석자 구분은 다시 실행 불가”를 구분한다.

6. 자동 후속 실행 목표와 현재 UI 상태를 구분해야 한다.
   - 제품 목표는 사용자가 기술 모드를 고르지 않아도 앱이 가능한 작업을 이어서 처리하는 것이다.
   - 현재 구현은 대화록 작성 후 참석자 구분과 기록 정리를 결과 화면에서 별도로 실행하는 구조다.
   - 1차 구현은 현재 구조를 유지하되 사용자 문구를 보강하고, 2차 구현에서 참석자 구분/기록 정리를 안전하게 자동 후속 작업으로 이어갈 수 있는지 검토한다.

## 자동 처리 흐름

사용자에게는 단순하게 보이고, 내부적으로는 단계별 저장을 한다.

```text
파일 선택
-> 길이/크기/저장 공간 확인
-> 자동 처리 계획 결정
-> 음성 추출
-> 대화록 구간별 작성
-> 대화록 먼저 저장
-> 참석자 구분 가능 여부 확인
-> 가능하면 참석자 구분 실행, 어렵다면 나중 실행 또는 제외
-> 기록 정리 가능 상태 표시
```

사용자에게는 다음처럼 보인다.

```text
대화록 작성 중
긴 영상이라 시간이 걸릴 수 있습니다.
완료된 구간은 저장하고 있어 중간에 중지해도 이어서 진행할 수 있습니다.
```

자동 처리의 단계별 목표:

- 1차: 대화록을 안정적으로 먼저 저장하고, 참석자 구분과 기록 정리 상태를 결과 화면에서 명확히 보여준다.
- 2차: 참석자 구분이 위험하지 않은 파일은 대화록 저장 후 백그라운드 후속 작업으로 자연스럽게 이어갈 수 있게 한다.
- 3차: 긴 참석자 구분과 긴 기록 정리는 내부 분할/부분 정리 기반으로 자동 진행한다.

1차에서는 “자동으로 다 끝낸다”를 과하게 약속하지 않는다. 대신 사용자는 대화록을 먼저 얻고, 다음에 무엇이 가능한지 바로 알 수 있어야 한다.

## 길이별 사용자 안내 기준

아래 기준은 사용자를 위한 안내 기준이다. 실제 내부 판단은 파일 크기, WAV 추정 크기, PC 상태까지 함께 본다.

| 길이 | 기본 동작 | 사용자 안내 |
| --- | --- | --- |
| 30분 이하 | 일반 자동 진행 | 일반적인 길이입니다. 대화록과 참석자 구분을 진행할 수 있습니다. |
| 30분~1시간 30분 | 대화록 우선, 참석자 구분 별도 상태 표시 | 긴 영상입니다. 대화록을 먼저 저장하고 참석자 구분은 이어서 진행합니다. |
| 1시간 30분~2시간 20분 | 긴 파일 자동 처리 | 시간이 오래 걸릴 수 있습니다. 완료된 구간을 저장하며 진행합니다. |
| 2시간 20분 이상 | 대화록 우선, 참석자 구분 신중 처리 | 대화록을 먼저 저장합니다. 참석자 구분은 PC 상태에 따라 오래 걸리거나 제외될 수 있습니다. |

사용자에게 수치 기준을 과하게 노출하지 않는다. 다만 상태 설명은 분명해야 한다.

## UI/UX 계획

### 1. 파일 선택 영역

현재 파일명, 크기, 길이는 표시된다. 여기에 긴 파일 안내를 한 줄 추가한다.

예시:

```text
긴 영상입니다. 대화록을 먼저 저장하고, 참석자 구분은 이어서 진행합니다.
```

상태별 문구:

- 30분 이하: `일반적인 길이입니다.`
- 30분~1시간 30분: `긴 영상입니다. 대화록을 먼저 저장합니다.`
- 1시간 30분 이상: `긴 영상입니다. 완료된 구간을 저장하며 진행합니다.`
- 참석자 구분 위험: `참석자 구분은 시간이 오래 걸리거나 나중에 실행될 수 있습니다.`

### 2. 분석 진행 화면

현재 경과 시간과 예상 시간이 있다. 이를 다음처럼 분리한다.

- 대화록 작성 예상 시간
- 이후 작업 상태
- 예상 정확도 수준

예시:

```text
대화록 작성 중
예상 시간: 약 35분
참석자 구분과 기록 정리는 대화록 저장 후 이어서 진행합니다.
```

예상 시간 문구 단계:

- 시작 직후: `예상 시간 측정 중`
- 초반 구간 완료 후: `대화록 작성 약 35분 남음`
- 대화록 완료 후: `대화록 저장 완료`
- 참석자 구분 중: `참석자 구분은 시간이 오래 걸릴 수 있습니다`
- 정리 중: `긴 회의라 기록을 나누어 정리하고 있습니다`

긴 파일에서는 진행 문구를 반복 설명하지 않고 상태만 바꾼다.

- `음성 준비 중`
- `대화록 작성 중`
- `대화록 저장 중`
- `참석자 구분 준비 중`
- `기록 정리 준비됨`

### 3. 중지/취소 안내

현재 중지와 취소가 분리되어 있다. 긴 파일에서는 중지의 의미를 더 명확히 한다.

```text
중지하면 완료된 구간을 남기고 나중에 이어서 진행할 수 있습니다.
취소하면 이번 진행 기록을 남기지 않습니다.
```

### 4. 결과 화면

현재 대화록, 참석자 구분, 기록 정리 상태가 분리되어 있다. 긴 파일 기준으로 상태 문구를 보강한다.

- 대화록: `완료`
- 참석자 구분: `대기`, `진행 중`, `제외됨`, `확인 필요`, `완료`
- 기록 정리: `준비됨`, `정리 중`, `모델 준비 필요`, `완료`

참석자 구분이 제외된 경우:

```text
대화록은 저장되었습니다. 이 파일은 길어 참석자 구분을 자동으로 실행하지 않았습니다.
필요하면 원본 음성을 보존한 상태에서 다시 시도할 수 있습니다.
```

### 5. 이어하기 화면

현재 이어하기 카드가 있다. 긴 파일에서는 다음 상태를 더 명확히 한다.

- `대화록 일부 저장됨`
- `대화록 완료, 참석자 구분 대기`
- `참석자 구분 중단됨`
- `같은 파일 필요`
- `이어하기 불가`

### 6. 설정 화면

현재 참석자 구분, 음성 파일 보존 옵션이 있다. 긴 파일 관점에서는 설명을 다듬는다.

권장 설명:

```text
내부 음성 파일을 남겨두면 대화록 작성 후 참석자 구분을 나중에 실행할 수 있습니다.
긴 영상에서는 이 옵션을 켜는 것이 좋습니다.
```

## 내부 저장 계획

사용자에게 직접 노출하지 않되, 아래 산출물은 안정적으로 남겨야 한다.

- 원본 업로드 정보
- 추출 WAV 경로와 길이
- chunk 목록
- chunk별 STT 결과
- 병합 대화록
- 참석자 구분 실행 전 상태
- 참석자 구분 결과
- 기록 정리 부분 결과
- 최종 결과 파일

긴 영상 시작 전 최소 점검:

- 선택한 파일 크기
- 추출 WAV 예상 크기
- chunk 파일 예상 크기
- output/temp 폴더가 있는 드라이브의 여유 공간
- 내부 음성 파일 보존 여부

디스크 여유 공간 1차 기준:

```text
필요 여유 공간 = 원본 파일 크기 + 예상 WAV 크기 * 2 + 1GB 안전 여유
```

이 기준은 정확한 저장량 예측이 아니라 분석 시작 전 명백한 실패를 줄이기 위한 보수적 안내다. 부족하면 분석을 시작하지 않고 “저장 공간을 확보한 뒤 다시 시도”를 안내한다.

RAM 여유는 1차에서 차단 조건보다 경고 조건으로 둔다. Windows 환경에서 신뢰도 높은 RAM 판단은 별도 시스템 상태 API가 필요하므로, 처음에는 긴 영상/참석자 구분 단계의 위험 안내와 자동 제외 기준을 우선 적용한다.

실패 후 재시작 기준:

- 같은 파일이면 완료된 STT 구간은 재사용한다.
- 대화록이 완료된 상태면 참석자 구분부터 다시 시도할 수 있다.
- 참석자 구분이 완료된 상태면 기록 정리만 다시 실행할 수 있다.
- 설정이 달라졌으면 기존 결과를 자동 재사용하지 않고 사용자에게 알려준다.

## 참석자 구분 장기 처리 계획

현재 참석자 구분은 긴 WAV에서 가장 위험한 단계다. 당장 사용자는 직접 파일을 자르지 않아야 한다. 내부적으로 다음 방향을 검토한다.

1. 먼저 대화록을 저장한다.
2. 참석자 구분은 별도 단계로 실행한다.
3. 너무 길거나 PC 상태가 부족하면 자동으로 제외하고 이유를 남긴다.
4. 다음 단계에서 10~20분 단위 내부 분할을 검토한다.
5. 분할 시 앞뒤 30~60초를 겹치게 처리한다.
6. 조각 사이 참석자 라벨 병합은 별도 품질 검증 후 적용한다.
7. 불확실한 짧은 끼어들기나 겹침 구간은 `참석자 확인 필요`로 표시한다.

## 긴 대화록 정리 계획

긴 대화록은 한 번에 정리하지 않는다.

```text
시간대별 부분 정리
-> 주제 후보 정리
-> 전체 요약
-> 주제별 정리
-> 참석자별 정리
```

사용자에게는 다음처럼 보이면 된다.

```text
긴 회의라 기록을 나누어 정리하고 있습니다.
완료된 정리는 먼저 확인할 수 있습니다.
```

## 사용자가 조치해야 하는 예외 상황

사용자에게 선택을 요청하는 경우는 아래로 제한한다.

| 상황 | 안내 | 선택 |
| --- | --- | --- |
| 저장 공간 부족 | 임시 파일을 만들 공간이 부족합니다. | 저장 공간 확보 후 다시 시도 |
| 원본 음성 없음 | 참석자 구분에 필요한 음성이 없습니다. | 다시 분석 또는 대화록만 사용 |
| 참석자 구분 반복 실패 | 이 PC에서는 참석자 구분이 어렵습니다. | 대화록만 사용, 다시 시도, 파일 나누기 |
| 앱 중지/닫기 | 완료된 구간을 남기고 중지할 수 있습니다. | 중지, 취소 |
| 정리 모델 없음 | 기록 정리 모델이 준비되지 않았습니다. | 모델 준비 후 정리 |

## 구현 순서

1. 긴 파일 사전 안내 문구 추가
   - 파일 선택 영역에 길이 기반 안내를 추가한다.
   - 개발 용어 없이 “대화록 먼저”, “참석자 구분은 이어서”로 표현한다.

2. 진행 화면 문구 정리
   - 예상 시간은 대화록 기준임을 유지한다.
   - 참석자 구분/기록 정리는 후속 단계로 짧게 표시한다.

3. 결과 화면 상태 문구 보강
   - 참석자 구분 제외/대기/실패/재시도 상태를 사용자 문구로 정리한다.

4. 이어하기 상태 문구 보강
   - “대화록 일부 저장됨”, “대화록 완료, 참석자 구분 대기”를 구분한다.

5. 저장 공간 사전 점검
   - 긴 영상에서는 임시 WAV와 chunk 파일이 생기므로 여유 공간을 확인한다.
   - 부족하면 분석 시작 전에 안내한다.

6. 참석자 구분 내부 분할 실험
   - 전체 WAV 방식과 window 방식의 품질, 시간, 메모리 사용량을 비교한다.
   - 바로 기본값으로 바꾸지 않는다.

7. 긴 대화록 부분 정리
   - 시간대별 부분 정리 checkpoint를 만들고 최종 정리로 합친다.

단계별 게이트:

- 1차 구현은 `MeetingWriter.tsx`, `MeetingHistory.tsx`, `Settings.tsx`의 사용자 문구와 상태 표시 보강을 중심으로 한다.
- 1차에서는 참석자 구분을 자동으로 새로 이어 실행하는 동작 변경을 넣지 않는다.
- 2차에서 디스크 여유 사전 점검 API와 UI 경고를 추가한다.
- 3차에서 참석자 구분 자동 후속 실행 여부를 결정한다.
- 4차 이후에만 참석자 구분 내부 분할과 긴 대화록 부분 정리를 구현한다.

이 게이트를 두는 이유는 사용자 안내 개선과 처리 파이프라인 변경을 한 번에 섞지 않기 위해서다.

## 검증 기준

길이별 샘플:

- 30분
- 1시간 30분
- 2시간 30분
- 가능하면 5시간

각 샘플에서 확인할 항목:

- 대화록이 먼저 저장되는가
- 중지 후 같은 파일로 이어서 진행되는가
- 참석자 구분이 위험할 때 사용자에게 이해 가능한 문구가 나오는가
- 참석자 구분 실패 후 대화록이 남는가
- 기록 정리는 대화록 완료 후 별도 실행 가능한가
- 임시 파일이 과도하게 남지 않는가
- 최종 TXT/JSON/HWPX/DOCX 내보내기가 깨지지 않는가

## 사용자 문구 원칙

쓰지 않을 표현:

- chunk
- checkpoint
- diarization
- waveform
- pyannote
- backend
- memory limit

사용할 표현:

- 구간
- 이어서 진행
- 완료된 내용 저장
- 참석자 구분
- 대화록 먼저 저장
- 시간이 오래 걸릴 수 있음
- 저장 공간 부족
- 원본 음성 필요

## 결론

긴 영상 처리는 사용자가 처리 방식을 고르는 기능이 아니라, 앱이 조용히 자동 적용해야 하는 안정성 기능이다. 사용자는 진행 상태와 필요한 조치만 알면 된다. 우선은 기존 UI의 파일 길이 표시, 예상 시간, 이어하기, 참석자 구분 별도 실행 흐름을 살리고, 긴 파일 전용 안내와 복구 상태 표시를 보강한다.
