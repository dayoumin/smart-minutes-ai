Codex collaboration guide

리뷰 요청을 덜 꼬이게 하는 관점 정리

사용자가 정확한 기술 용어를 몰라도 괜찮습니다. 중요한 것은 “무엇을 결정하려는지”와 “어느 깊이까지 확인해야 하는지”를 Codex가 놓치지 않게 요청하는 것입니다.

큰 관점 작은 관점 agent 리뷰 검증과 커밋 프로젝트 공통
요약 용어 요청 문장 확인표 공통 사용 리뷰 깊이 agent 빠지는 항목

한눈에 보는 사용법

1. 목표를 말한다 무엇을 만들고, 어디에 배포하고, 누가 쓰는지 먼저 알려줍니다.
2. 관점을 나눈다 큰 관점은 목표와 리스크, 작은 관점은 파일과 실행을 확인합니다.
3. 끝 지점을 정한다 진단만, 수정까지, 커밋까지, 푸시까지 중 어디까지 할지 정합니다.
4. 검증을 남긴다 실행 명령, 확인 결과, 남은 리스크를 짧게 남기면 다음 작업이 쉬워집니다.

핵심 구분

구조 설계 코드, 모듈, 데이터, 파일의 배치 문제입니다. 예를 들면 backend, desktop-app, scripts, models를 어떻게 나누는지, 설정 파일을 어디에 둘지, 모델 원본과 배포본 경로를 어떻게 분리할지 보는 관점입니다.
시스템 설계 실행되는 전체 시스템의 동작 문제입니다. 예를 들면 Tauri 앱, sidecar 백엔드, 모델 폴더, Ollama, 출력 파일, 포터블 배포, 장애 복구가 어떤 순서로 이어지는지 보는 관점입니다.
큰 관점 설계만 보는 것이 아니라 “지금 목표에 맞게 가져가도 되는 상태인가?”를 봅니다. 제품 목표, 회사 PC 배포, 용량, 검증 수준, 남은 리스크, 다음 행동 순서까지 함께 봅니다.
작은 관점 실제 코드, 설정, 경로, 명령, 로그, 테스트 결과가 맞는지 확인합니다. 버튼 하나, API 하나, 모델 경로 하나, 빌드 스크립트 하나처럼 구체적인 실패 가능성을 봅니다.

구조 설계와 시스템 설계는 큰 관점 안에 포함될 수 있지만, 큰 관점과 같은 말은 아닙니다. 큰 관점은 “설계가 맞나?”를 넘어 “이 상태로 목표를 달성할 수 있나?”까지 묻는 표현입니다.

왜 큰 관점과 작은 관점을 같이 요청하나

큰 관점만 보면 실행 명령, 경로, 환경변수 같은 작은 문제가 빠질 수 있습니다. 작은 관점만 보면 당장 고친 코드는 맞아도 배포 목표나 회사 PC 상황과 어긋날 수 있습니다.

그래서 “큰 관점 + 작은 관점” 요청은 Codex에게 다음 두 가지를 동시에 하라는 뜻으로 쓰면 좋습니다.

  1. 먼저 목표, 사용자 흐름, 배포 환경, 리스크를 기준으로 방향이 맞는지 본다.
  2. 그 다음 파일, 코드, 설정, 명령, 테스트로 실제로 맞는지 확인한다.

Codex에게 요청할 때의 추천 문장

짧게 요청하고 싶을 때는 아래처럼 말하면 충분합니다.

큰 관점과 작은 관점으로 나눠서 리뷰해줘. 큰 관점에서는 목표, 배포 가능성, 남은 리스크, 다음 순서를 봐줘. 작은 관점에서는 코드, 설정, 경로, 명령, 테스트가 실제로 맞는지 확인해줘. 바로 수정이 필요한 것은 수정하고, 판단이 필요한 것은 이유와 선택지를 정리해줘.

agent 리뷰를 명시하고 싶을 때

agent로 큰 관점 리뷰와 작은 관점 리뷰를 나눠서 해줘. 큰 관점 agent는 제품 목표, 회사 PC 배포, 모델 용량, 운영 리스크를 봐줘. 작은 관점 agent는 코드, 설정, 모델 경로, 빌드 스크립트, 테스트 가능성을 봐줘. 리뷰 후 바로 고칠 것은 고치고, 커밋/푸시 전 확인할 것을 정리해줘.

아직 수정하지 말고 진단만 하고 싶을 때

아직 수정하지 말고 상황진단만 해줘. 큰 관점에서는 지금 방향이 맞는지, 회사에 가져가도 되는지 봐줘. 작은 관점에서는 실제 파일과 명령 기준으로 문제가 될 지점을 찾아줘. 수정이 필요하면 바로 고치지 말고 우선순위와 이유만 정리해줘.

리뷰 범위별 확인표

관점 주요 질문 이 프로젝트에서의 예시
구조 설계 파일과 책임이 올바른 위치에 있는가? 재빌드용 모델 원본은 프로젝트 루트 models, 최종 실행본은 releases\lmo_audio\models로 분리되어 있는가?
시스템 설계 앱 실행부터 분석 완료까지 흐름이 이어지는가? Tauri 프론트, sidecar 백엔드, 모델 상태 API, 분석 API, 출력 파일 생성이 같은 경로 기준으로 동작하는가?
큰 관점 지금 가져가도 되는 상태인가? 회사 PC에서 재빌드 가능한가, Qwen 제외로 용량이 줄었는가, 필요한 모델과 빌드 도구가 문서화됐는가?
작은 관점 구체적으로 실패할 부분은 없는가? release_portable.ps1, configs/asr-models.json, 모델 폴더, API health, 프론트 URL이 실제로 맞는가?

요청할 때 같이 주면 좋은 정보

  • 이번 목표: 예를 들어 “회사 PC에서 pull 후 실행 파일을 다시 만들기”
  • 현재 상태: 예를 들어 “웹에서 개발하고 배포는 데스크톱 앱으로 한다”
  • 걱정되는 점: 예를 들어 “Qwen 때문에 용량이 커진다”, “회사 PC에서 빌드가 자주 꼬였다”
  • 원하는 행동: 예를 들어 “진단만”, “바로 수정”, “agent 리뷰 후 커밋”, “푸시까지”
  • 검증 기준: 예를 들어 “프론트 접속”, “API health”, “모델 상태”, “실제 오디오 90초 분석”

Codex가 이렇게 움직이면 좋다

좋은 진행 순서

  1. 먼저 현재 목표와 배포 환경을 짧게 재확인한다.
  2. 큰 관점에서 가져가도 되는 상태인지 판단한다.
  3. 작은 관점에서 실제 파일, 경로, 명령, 로그를 확인한다.
  4. 명확한 문제는 바로 고치고, 애매한 선택은 이유와 선택지를 말한다.
  5. 검증 결과와 남은 리스크를 정리한다.

피해야 할 진행

  • 큰 관점 요청을 구조 설계만 보라는 뜻으로 좁게 해석하는 것
  • 작은 관점 확인 없이 “될 것 같다”고 말하는 것
  • 회사 PC 배포라는 목표를 빼고 로컬 개발 성공만 확인하는 것
  • 백엔드 포트와 프론트 URL처럼 사용자가 실제로 접속해야 하는 대상을 섞는 것

한 문장으로 정리

사용자가 “큰 관점, 작은 관점으로 봐줘”라고 말하면 Codex는 “목표에 맞게 가져가도 되는지”와 “실제 파일과 실행이 맞는지”를 나누어 확인해야 합니다. 구조 설계와 시스템 설계는 그 안에서 필요한 만큼 포함해서 보면 됩니다.

다른 프로젝트에서도 쓰는 방법

이 문서는 특정 프로젝트 전용 체크리스트가 아니라, Codex에게 리뷰를 요청하는 방법을 정리한 공통 가이드로 쓸 수 있습니다. 프로젝트가 바뀌면 예시만 바꾸고, 관점 구분과 요청 방식은 그대로 가져가면 됩니다.

이 프로젝트에서도 큰 관점과 작은 관점으로 리뷰해줘. 큰 관점에서는 목표, 사용자, 배포/운영 환경, 비용, 보안, 장애 가능성, 다음 순서를 봐줘. 작은 관점에서는 실제 파일, 코드, 설정, 데이터 경로, 실행 명령, 테스트 결과를 확인해줘. 프로젝트 특수성이 있으면 먼저 파악하고, 명확한 문제는 수정해줘.

리뷰 깊이 선택

요청 표현 Codex가 해야 할 일 적합한 상황
빠르게 봐줘 명백한 위험, 실행 여부, 다음 행동만 확인한다. 시간이 없고 큰 방향만 확인하면 될 때
큰 관점으로 봐줘 목표, 배포, 운영, 사용자 영향, 남은 리스크를 중심으로 판단한다. 지금 진행해도 되는지 결정해야 할 때
작은 관점으로 봐줘 파일, 코드, 설정, 경로, 명령, 테스트처럼 구체적인 실패 지점을 확인한다. 실행 오류, 빌드 오류, UI 동작 오류를 잡을 때
꼼꼼히 리뷰해줘 큰 관점과 작은 관점을 모두 보고, 검증과 수정까지 이어간다. 배포, 커밋, 회사 이관, 중요한 기능 변경 전

결과물까지 지정하기

리뷰 요청에서 가장 자주 빠지는 것은 “검토 후 무엇을 남길지”입니다. Codex가 진단만 하면 되는지, 바로 수정해야 하는지, 커밋과 푸시까지 해야 하는지 함께 말하면 진행이 덜 흔들립니다.

리뷰 후 결과물은 이렇게 정리해줘. 1. 지금 바로 고친 것 2. 검증한 명령과 결과 3. 남은 리스크 4. 내가 다음에 해야 할 일 5. 커밋/푸시가 필요하면 커밋 메시지와 이유

“수정하지 말고 진단만”, “바로 수정”, “커밋 전까지만”, “푸시까지”처럼 끝 지점을 정해주면 Codex가 중간에서 멈추거나 과하게 진행할 가능성이 줄어듭니다.

agent를 쓸 때와 안 쓸 때

agent를 쓰면 좋은 경우 범위가 넓고 동시에 볼 관점이 다를 때 좋습니다. 예를 들면 제품/배포 리스크와 코드/테스트 리스크를 분리해서 보고 싶을 때, 또는 한 agent는 큰 관점, 다른 agent는 작은 관점을 맡기고 싶을 때입니다.
agent 없이 바로 하는 경우 단일 파일 수정, 명확한 오류, 간단한 실행 확인처럼 범위가 좁을 때는 바로 작업하는 편이 빠릅니다. agent를 쓰면 오히려 확인 시간이 늘어날 수 있습니다.
agent를 써야 할지 먼저 판단해줘. 범위가 넓으면 큰 관점 agent와 작은 관점 agent로 나눠서 리뷰하고, 범위가 좁으면 바로 직접 확인하고 수정해줘.

프로젝트 정보 카드

다른 프로젝트에서 처음 리뷰를 요청할 때는 아래 정보를 짧게 붙이면 Codex가 훨씬 빨리 맥락을 잡습니다.

프로젝트 정보: - 목표: - 사용자: - 현재 실행 방식: - 배포/운영 방식: - 중요한 폴더: - 외부 의존성: - 절대 망가지면 안 되는 흐름: - 지금 걱정되는 점: - 원하는 종료 지점: 진단만 / 수정까지 / 커밋까지 / 푸시까지

자주 빠지는 검토 항목

  • 사용자가 실제로 여는 화면 또는 실행 파일이 무엇인지
  • 프론트엔드 주소와 백엔드 API 주소가 섞이지 않았는지
  • 로컬 개발 경로와 배포 후 실행 경로가 다른지
  • 대용량 파일, 모델, 비밀키, 캐시가 Git에 올라가지 않는지
  • 새 PC에서 필요한 설치 도구와 환경변수가 문서화되어 있는지
  • 검증이 “빌드 성공”에서 끝나지 않고 실제 사용자 흐름까지 확인됐는지
  • 나중에 다시 볼 사람이 이해할 수 있도록 todo, README, 체크리스트 중 한 곳에 남겼는지
