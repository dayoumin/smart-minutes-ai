# 대화록 사용자 수정본 설계

## 목표

- STT 원본과 화자 정렬 원본은 보존한다.
- 사용자는 화면에서 대화록을 직접 수정할 수 있다.
- 수정 결과는 사용자 수정본으로 따로 저장한다.
- 내보내기, 주제별 정리, 참석자별 정리는 사용자 수정본을 기준으로 다시 만들 수 있어야 한다.

## 원칙

1. 원본을 덮어쓰지 않는다.
2. 사용자 수정본과 원본을 구분해서 저장한다.
3. 언제든 원본으로 되돌릴 수 있어야 한다.
4. 수정 후 어떤 결과가 사용자 수정본 기준인지 명확해야 한다.

## 권장 데이터 구조

회의록 저장 구조에 아래 필드를 추가하는 방향을 권장한다.

- `segments`
  - 원본 정렬 세그먼트
- `displaySegments`
  - 화면 표시용 병합 세그먼트
- `editedDisplaySegments`
  - 사용자가 수정한 표시용 대화록
- `transcriptEditMeta`
  - 수정 시각, 수정 여부, 수정본 기준으로 정리를 다시 만들었는지 등의 메타데이터

예시:

```json
{
  "segments": [],
  "displaySegments": [],
  "editedDisplaySegments": [],
  "transcriptEditMeta": {
    "edited": true,
    "editedAt": "2026-05-12T16:10:00+09:00",
    "summaryOutdated": true,
    "topicSectionsOutdated": true,
    "speakerContextOutdated": true
  }
}
```

## 화면 동작

### 1. 기본 표시

- `editedDisplaySegments`가 있으면 그걸 우선 표시
- 없으면 `displaySegments`
- 그것도 없으면 fallback 대화록

### 2. 편집 방식

- 세그먼트/문단 단위 `textarea` 편집을 우선 검토
- 한 번에 전체 긴 문서를 자유 편집하는 방식보다, 현재 화면 구조를 유지한 채 각 발화 블록을 수정하는 쪽이 안전하다

### 3. 사용자 액션

- `편집`
- `저장`
- `원본으로 되돌리기`
- `이 수정본으로 다시 정리`

## 정리/내보내기와의 관계

- TXT/MD/DOCX/HWPX 내보내기는 사용자 수정본이 있으면 그 기준을 우선 사용
- `주제별 정리`, `참석자별 정리`는 사용자 수정본이 있으면 "정리 갱신 필요" 상태로 바꾼다
- 사용자가 원하면 수정본 기준으로 다시 생성한다

## 주의할 점

1. 원본 세그먼트의 시간 정보는 유지해야 한다.
2. 수정본은 시간축 편집보다 텍스트 수정이 먼저다.
3. 참석자 이름 수정과 대화록 본문 수정은 별도 상태로 본다.
4. 사용자가 수정한 뒤 자동으로 기존 요약을 신뢰하면 안 된다.

## 구현 우선순위

1. `editedDisplaySegments` 저장 구조 추가
2. 화면에서 세그먼트 단위 편집 UI 추가
3. 내보내기에서 수정본 우선 사용
4. 수정 후 정리 결과 stale 상태 표시
5. 수정본 기준 재정리 버튼 연결
