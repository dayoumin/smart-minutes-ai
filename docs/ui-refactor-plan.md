# 회의 요약 / 대화록 UI 리팩토링 계획

작성일: 2026-05-13

## 목적

- `회의 요약`, `대화록`, `새 회의록 작성` 화면의 정보 계층을 단순화한다.
- 긴 설명문과 중복 상태 문구를 줄인다.
- 반복 UI를 공통 컴포넌트로 분리해 이후 카피/상태 규칙을 한곳에서 관리한다.
- 현재 실행 중인 긴 파일 분석이 끝난 뒤, 안전하게 순차 적용할 수 있게 범위를 미리 고정한다.

## 현재 문제

1. `새 회의록 작성` 첫 화면에서 복구 패널 비중이 커서 기본 작업보다 예외 흐름이 먼저 읽힌다.
2. `정리` 영역에서 결과, 상태, 실행 버튼, 전제조건 안내가 한 화면에 섞여 있다.
3. `참석자별 정리`는 읽기 화면인데 필터/접기/이름 연결 제어가 많아 밀도가 높다.
4. 상태 문구가 배너, 카드 메타, 인라인 노트에 나뉘어 있어 같은 의미가 반복된다.
5. 버튼 언어가 아직 완전히 통일되지 않았다.
6. resume 선택 상태인데도 기본 CTA가 `분석 시작`으로 고정되어 있어 현재 맥락을 반영하지 못한다.
7. `미완료 분석 기록` 배지와 resume 카드가 semantic status token 대신 임시 muted/background 조합에 기대어 상태 의미가 약하다.

## 리팩토링 원칙

- 결과를 먼저 보여주고, 상태는 짧게 붙인다.
- 전제조건 설명은 버튼 옆이 아니라 섹션 레벨에서 한 번만 보여준다.
- 액션 버튼은 짧게 유지하고, 상태 표시는 버튼이 아니라 메타나 아이콘에서 처리한다.
- helper text는 섹션당 한 줄만 허용한다.
- 같은 의미의 상태는 같은 컴포넌트와 같은 카피 규칙을 쓴다.

## 공통 컴포넌트 후보

### 1. `SummaryActionCard`

대상:
- `전체 요약`
- `주제별 정리`
- `참석자별 정리`

역할:
- 제목
- 완료 아이콘
- 상태 메타
- 액션 버튼
- disabled 이유는 카드 바깥의 섹션 안내로 위임

현재 후보 위치:
- `desktop-app/src/MeetingHistory.tsx`의 `정리` 3개 카드

예상 props:
- `title`
- `status`
- `outdated`
- `actionLabel`
- `onAction`
- `disabled`
- `showAction`

### 2. `InlineStateNote`

대상:
- `정리 기능은 ... 사용할 수 있습니다`
- `주제별 정리를 하면 아래에 ...`
- `현재 화면은 수정본 대화록입니다`
- `이름을 연결하면 ... 반영됩니다`

역할:
- 짧은 보조문만 렌더링
- 성공/주의/중립 톤 최소화

현재 후보 위치:
- `desktop-app/src/MeetingHistory.tsx`
- `desktop-app/src/MeetingWriter.tsx`

### 3. `ResumeDraftSection`

대상:
- `진행 중이던 분석 기록`
- `이전 분석 기록`

역할:
- 섹션 제목
- 보조문
- 카드 리스트

분리 이유:
- 현재 `MeetingWriter.tsx` 안에서 `active`와 `resumable` 렌더링 분기가 길다.
- 이후 “기본 폼 우선, 복구 패널 접기/축소” 조정이 쉬워진다.

### 4. `ResumeDraftCard`

대상:
- resume draft 단일 카드

역할:
- 제목
- 상태/파일명/시간
- 선택 상태
- 액션 버튼

주의:
- `준비됨` 같은 상태는 버튼 문구가 아니라 카드 배지로 옮기는 방향 검토
- `active`, `resumable`, `selected` 상태를 색상과 배지로 구분하고, 버튼은 동작만 남기는 방향 검토

### 5. `SpeakerLabelPanel`

대상:
- 대화록 탭의 화자 이름 연결 패널

역할:
- 안내 문구
- dirty state
- speaker 입력 리스트

분리 이유:
- 대화록 편집 액션과 화자 이름 연결 액션을 분리해 읽기 쉽게 만든다.

### 6. `SpeakerSummaryList`

대상:
- 참석자별 정리 카드 리스트

역할:
- 전체/선택 필터
- 카드 리스트
- 접기 상태

주의:
- 2차에서는 제어 수를 줄이는 방향으로 재구성
- 이름 연결 버튼은 섹션 밖 상위 액션으로 이동 검토

## 화면별 적용 순서

### 1단계: `MeetingHistory.tsx`

우선 이유:
- 현재 가장 복잡하고, 상태/카피 중복도 가장 많다.

작업:
1. `SummaryActionCard` 도입
2. `InlineStateNote` 도입
3. `SpeakerLabelPanel` 분리
4. `SpeakerSummaryList` 분리 초안
5. 빈 상태 문구 통일

검증:
- `회의 요약` 탭에서 상태/버튼/보조문이 한눈에 구분되는지
- `대화록` 탭에서 편집과 이름 연결이 서로 덜 섞이는지

### 2단계: `MeetingWriter.tsx`

우선 이유:
- 새 회의록 작성 흐름이 기본 사용 경로이기 때문

작업:
1. `ResumeDraftSection` 분리
2. `ResumeDraftCard` 분리
3. `진행 중이던 분석 기록` / `이전 분석 기록` 시각적 비중 축소
4. 기본 CTA를 상태 기반으로 분기
   - 기본: `분석 시작`
   - resumable 선택 + 파일 일치: `이어받기`
   - active 충돌: 시작 비활성 + 상태 확인 유도
5. `미완료 분석 기록` 배지와 resume 카드에 semantic status token 적용
6. completion/status banner 문구 정리
7. helper text 총량 축소

검증:
- 새 회의록 작성 폼이 먼저 읽히는지
- 복구 흐름은 필요할 때만 눈에 들어오는지
- resume 선택 시 버튼이 `분석 시작`이 아니라 현재 맥락에 맞는 동작으로 읽히는지
- `미완료 분석 기록`, `진행 중이던 분석 기록`, `이전 분석 기록`의 상태 차이가 색상과 배지로 구분되는지

## 상태/카피 정리 계획

### 상태 메타

- 완료: 아이콘만
- 진행 중: `정리 중`, `저장 중`, `확인 중`
- 주의: `수정 필요`, `원본 필요`, `대화록 필요`
- 불가 조건: 섹션 하단에 한 번만 설명

### 상태 색상

- `진행 중`: `info` 또는 별도 진행 상태 토큰
- `이어받기 가능`: `neutral` 또는 `info` 계열 배지
- `사용자 취소` / `실패`: 같은 muted 표면이 아니라 상태 차이가 보이는 배지 사용
- 사이드바 `미완료 분석 기록`은 단순 `bg-muted/30` 대신 semantic status pill 또는 전용 summary badge로 통일

### 배너

- 완료 배너는
  - 제목
  - 핵심 한 줄
  - 필요 시 resume note 1줄
  로 제한

- 진행 패널은
  - 현재 단계
  - 경과 시간
  - 남은 시간
  - 취소
  만 유지하고 설명문은 최소화

### 빈 상태

- 형식 통일:
  - `주제별 정리 내용이 없습니다. 대화록을 확인해 주세요.`
  - `참석자별 정리 내용이 없습니다. 화자 구분을 확인해 주세요.`
  - `대화록 내용이 없습니다.`

## 테스트 계획

긴 파일 분석 종료 후 적용한다.

### 수동 확인

1. `회의 요약`
   - 정리 카드에서 완료/진행/불가 상태 구분
   - 주제별 정리 / 참석자별 정리 표시 밀도

2. `대화록`
   - 편집 버튼, 저장 버튼, 이름 연결 패널 관계
   - 빈 상태 / 수정본 안내 톤

3. `새 회의록 작성`
   - 복구 패널보다 기본 입력 폼이 먼저 읽히는지
   - active/resumable 카드의 정보 밀도
   - 진행 패널과 완료 배너 길이

### 자동 검증

- `corepack pnpm --dir desktop-app typecheck`
- `corepack pnpm --dir desktop-app lint`
- `corepack pnpm --dir desktop-app build`

추가 시뮬레이션 후보:
- resume 카드 표시/선택 흐름
- 정리 카드 상태 표시 회귀

## 구현 금지선

- 현재 긴 파일 분석 중에는 `MeetingWriter.tsx`, `MeetingHistory.tsx`, `backend/main.py`를 수정하지 않는다.
- 분석 종료 후 브라우저 상태를 새로고침하고 나서 UI 리팩토링을 시작한다.
- 상태 문구를 줄이더라도 resume/정리/대화록 편집 기능 의미는 유지한다.
